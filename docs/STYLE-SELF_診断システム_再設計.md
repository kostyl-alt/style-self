# STYLE-SELF 診断システム 再設計(アプローチ2)

作成日: 2026-05-15
ステータス: 設計フェーズ。実装前に判断を確定する。
オーナーの判断: 8パターン体制を廃止し、AI が世界観を「その人だけのもの」として毎回構築する形に移行する。リスクを認識した上で一気にやる。

---

## 0. なぜこの決断をしたか(ビジョンとの対応)

オーナーのビジョン:
- 「新しい自分を知れる診断」
- 「自分でも気づいていなかった好きな雰囲気を言語化」
- 「他の人にはない、自分だけの世界観」
- 「単なるファッション診断ではなく、服を通して自分を知るアプリ」

8パターンの枠だと:
- 16問の回答が枠に押し込まれる
- 同じパターンに分類された人は同じ結果
- 「あなただけの」体験にならない

アプローチ2 の世界:
- 16問の回答を AI が全体として読み解く
- パターンに収めず、その人固有の世界観を言語化する
- 影響源・色・素材・ブランドも、その人の回答から導出する
- 結果として無限の世界観が生まれうる

## 2. 設計の核心: AI に何をさせるか

新しい /api/ai/analyze の流れ:

```
INPUT:
  16問の回答(answers)
  ユーザーの自由記述(Q15)
  避けたいもの(Q16)

ステップ1: Knowledge OS から情報取得(並列)
  - get_categories で 33カテゴリのツリー
  - get_influences で全影響源の概要(20件全部、または絞り込み)
  - get_decision_rules で判断ルール(上位)

ステップ2: AI 呼び出し(1回目: 世界観の言語化)
  Claude に渡すもの:
    - 16問の回答 全部
    - Knowledge OS の情報
    - 「このユーザーの世界観を言語化してください」というプロンプト
  AI が返すもの:
    - worldview_name(世界観名・固有の言葉)
    - keywords(この世界観を表す3〜5語)
    - selected_influences(関連する影響源 5人 + 理由)
    - selected_categories(関連カテゴリ slug)

ステップ3: AI 呼び出し(2回目: 13項目の詳細生成)
  Claude に渡すもの:
    - ステップ2 の出力
    - 選ばれた影響源の詳細データ(get_influences で個別取得)
    - 「この世界観に基づいて13項目を生成してください」
  AI が返すもの:
    - 13項目全部(無意識の傾向、なりたい自分、色、素材、ブランド、etc)

ステップ4: バリデーション + 保存
  - 必須項目チェック
  - DB に保存(worldview_profiles に新形式で)
  - 商品マッチング用のキーワード抽出
```

## 3. AI コール構成の判断

### なぜ2回に分けるか

1回で全部出させると:
- プロンプトが巨大になりすぎる(33カテゴリ + 20影響源 + 16問回答 + 指示)
- 出力 JSON も巨大で、JSON parse 失敗リスクが上がる
- AI が「考えながら出力」できない

2回に分けると:
- 1回目: 「どんな世界観か、どの影響源が関連するか」を判断
- 2回目: 確定した世界観 + 関連影響源だけ深く知って、詳細を生成
- 各 AI コールが「思考の単位」として明確
- レイテンシは増えるが、品質が上がる

### コストとレイテンシ

```
従来(8パターン): AI コール 1回 = 約3〜8秒
アプローチ2: AI コール 2回 = 約6〜15秒

- 診断は1回で終わる体験なので、+5〜10秒は許容範囲
- AI 料金は2倍弱
- でも「あなただけの診断」の価値があるなら、コストに見合う
```

## 4. 段階的実装(一気にやるが、安全な順序で)

「一気に」と決めたが、実装は安全な順序で進める。**3つのフェーズに分けて、各フェーズで動作確認しながら進む**:

### フェーズA: 新しい AI 呼び出しを並行で実装(既存は壊さない)

```
1. 新エンドポイント app/api/ai/analyze-v2/route.ts を新規作成
2. アプローチ2 の流れを実装
3. 既存の analyze は触らない
4. /admin から analyze-v2 をテスト
5. 動作確認: 16問の回答に対して、どんな診断結果が出るか
6. オーナーが結果を見て品質判断
```

このフェーズの良さ:
- 既存ユーザーへの影響ゼロ
- 既存 /api/ai/analyze は動き続ける
- 新システムを並行で育てられる

### フェーズB: 移行(切り替え)

```
1. フェーズAで品質が確認できたら、analyze-v2 を analyze に昇格
2. 既存 analyze を analyze-legacy にリネーム(保険として残す)
3. 8パターンの定数も残す(全部消すと analyze-legacy が動かなくなる)
4. UI を新形式に対応
5. 商品マッチングロジックを新形式に対応
```

### フェーズC: 旧コード削除

```
1. analyze-legacy が3〜7日エラーなく動かないことを確認
2. 旧 worldview-patterns.ts などを削除
3. matchWorldview() / applyPatternToResult() 削除
4. diagnosis_sessions.pattern_id をどうするか判断
   (deprecated カラムにする / 削除 / 互換維持)
```

## 5. 既存ユーザーの過去診断データの扱い

選択肢:
- **案a**: 過去診断は8パターン形式のまま保持。再診断で新形式に上書き
- **案b**: マイグレーションで全て新形式に変換(AI 再実行が必要・コスト大)
- **案c**: 過去診断は読めるが編集不可、新規診断のみ新形式

推奨: **案a**。
理由:
- データを壊さない
- ユーザーが再診断したら新形式に切り替わる
- 移行期の負荷が小さい

## 6. 商品マッチングの新しい仕組み

現状:
```
パターンの coreTags ↔ product.worldview_tags でスコアリング
+40点(共通項あり)
```

新形式の選択肢:
- **案X**: AI 出力に「product_keywords」を含めて、それで worldview_tags をマッチ
- **案Y**: AI 出力の影響源 tags(マルジェラなら minimal/deconstruction)で
  worldview_tags をマッチ
- **案Z**: 商品マッチングも別 AI 呼び出しにする(コール3回目)

推奨: **案X + 案Y のハイブリッド**。
AI に最終出力で `worldview_tags`(英語スラッグ配列)を返させる。
それを既存の product_match.ts のロジックに流す。
既存ロジック自体は壊さない。

## 7. リスクと緩和策

| リスク | 緩和策 |
|--------|--------|
| AI が品質の低い世界観を生成 | フェーズAで並行運用してオーナーがレビュー。品質基準を満たすまで本番化しない |
| 同じユーザーが2回診断したら違う結果 | プロンプトに temperature=0.3 程度を設定。完全な再現性は犠牲にするが、ある程度安定 |
| AI コストとレイテンシ増 | 2回コールでも合計15秒以内に抑える。失敗時のリトライ戦略を持つ |
| 過去診断との互換性 | 案a(過去は8パターンのまま温存) |
| 商品マッチングが壊れる | worldview_tags 出力を維持して既存ロジックに流す |
| Knowledge OS への依存度が増す | フェーズAで Knowledge OS が落ちた時の挙動を設計 |
| 8パターン依存箇所が大量にある | フェーズB の前に全箇所洗い出し(Claude Code に依頼) |
