## § 1. ★ オーナー verbatim 記録(★ 改変禁止・全文保全)

```
今の出力は、

* 真似したくなる強さがない
* 自分が着たいと思える魅力がない
* リアル試着としても弱い
* ムードボードの核が服に変換されてない
* 説明が長いのに、体験は弱い
って状態。

つまり問題は 「もう少しプロンプトをうまくする」じゃなくて、何を作らせるかの定義がズレてる こと。今のAIに求めるものを変える

今はたぶん、
ムードボード → 解説 → 無難なコーデ → スタジオ風画像 になってる。

でも本当に欲しいのは、
ムードボード → 欲しくなる方向性 → 真似したくなるコーデ → 現実感のある着用イメージ → 修正対話

今の何がダメか
スクショの感じだと、ダメな理由はかなり明確。

1. 服が"安全すぎる"
AIが無難にまとめてる。 だから、
* ちょっとモード
* ちょっと中性的
* ちょっと不穏
くらいで止まってる。
でもあなたが欲しいのは、 「その世界観ならではの服」 でしょ。

2. 画像が"欲しくなる画像"じゃない
今の画像は、
* スタジオっぽい
* 棒立ち
* ECモデルっぽい
* 人として魅力が薄い
* 自分が着る想像も弱い
だから 「真似したい」 にも 「自分が着たい」 にも行かない。

3. 出力が"読ませる設計"になってる
長い説明を読んで、下に1枚画像。 これはファッションの体験として弱い。
ファッションで欲しいのは、 一目で「これいい」「これ違う」が分かること。

4. ムードボードが"服"に翻訳されてない
ムードボードの画像から、
* 何を核にするか
* 何を日常化するか
* 何を小物に逃がすか
* 何をシルエットに落とすか
が弱い。
だから最終的に、 どこかで見たような服 になる。

今の「着用イメージ」は、結局 架空の服を架空の人に着せてる風 だから、購買にも真似にも繋がりにくい。今回の出力も、文章としてはかなり構造化されてるし、JSONやquickActionsまで出ていて前より進化してる。でも、内容を見ると「セミシアー黒ニット」「エナメルブラ透け」「IDタグ」みたいな架空コーデを作って、それを画像化している。これは購買判断というより、まだ"世界観のビジュアル遊び"に近い。

あなたの言う通り、商品を着れなきゃダメ。
つまり、根本方針はこれ。
画像生成で服を作るな。 実在商品を選ばせて、それを着せろ。

いま切り替えるべき構造
今の流れはたぶんこれ。
ムードボード
↓
AIが架空コーデを考える
↓
AIが架空の着用画像を作る
↓
なんかリアルじゃない
↓
買えない

これをやめる。
本命はこれ。
ムードボード
↓
世界観を分析
↓
実在の商品候補を探す
↓
ユーザーが商品を選ぶ
↓
その商品画像を使って試着プレビュー
↓
買うか判断

方針を変更します。
現状の画像生成は、架空の服・架空の人物・架空の背景を生成しているため、リアル試着にも購買判断にもつながっていません。
ユーザーが欲しいのは「AIが作ったファッション画像」ではなく、「実在する商品を自分または近い体型モデルが着た時の判断材料」です。
したがって、STYLE SELFでは画像生成を主役にしません。
画像生成は一旦「世界観ルック確認」用途に限定し、購買導線の本命は「実商品試着プレビュー」に切り替えます。

新しい方針：
1. AIは架空の服を作らない
2. まず実在する商品候補を出す
3. 商品画像を使って着用プレビューを作る
4. 購買判断に使える表示にする
5. 生成画像は「買える商品」と必ず紐づける

新しいフロー：
1. ムードボード分析
2. Knowledge OSから判断ルール取得
3. コーデ方向性を作る
4. その方向性に合う実在商品候補を出す
5. 各商品に「なぜ合うか」を表示
6. ユーザーが商品を選ぶ
7. FASHN等のtry-on APIには、promptだけではなく model_image + garment_image を渡す
8. ユーザー本人写真または近い体型モデル画像に、その実商品を着せる
9. 着用プレビューを表示
10. 購入ページへ遷移

UI上のボタンを分ける：
- 世界観ルックを見る：架空画像でも可。雰囲気確認用。
- 商品を探す：実在商品の候補を出す。
- この商品を試す：商品画像を使って着用プレビュー。
- 近い体型で見る：本人写真なしで近似モデルに着せる。
- 自分で試す：本人写真を使う。明示同意必須。

重要：
「ムードボード → 架空コーデ → 架空画像」は購買に繋がらないため、主導線にしない。
購買導線は必ず「ムードボード → 実商品 → 試着プレビュー → 購入」にする。

画像生成の扱い：
- 今のFASHN画像生成は、世界観ルック確認用として残す
- ただし購入判断用とは明確に分ける
- 購入判断用では、必ず実商品画像を入力に使う
- 生成画像の下には必ず対応する商品名、価格、購入リンク、サイズ情報を表示する

今後の優先順位：
1. 架空画像生成の改善より、商品候補表示を優先
2. 商品画像を使ったtry-on連携を優先
3. ユーザー本人写真は後回しでもよい
4. 最初は近い体型モデル + 実商品画像で検証する
5. 本人写真を使う場合は同意UIと削除導線を必ず作る

今後の機能名も変えた方がいい
「着用イメージ」だとまだ曖昧。
こう分ける。

1. 世界観ルック
これは今の架空画像。 目的は「雰囲気を見る」だけ。

2. 商品試着プレビュー
実商品を着せる。 目的は「買う判断」。

3. 近い体型で見る
本人写真なし。 目的は「失敗を減らす」。

4. 自分で試す
本人写真あり。 目的は「一番リアルに近づける」。

この分け方が大事。

かなり重要な判断
今はもう、画像のクオリティを上げる方向に時間を使いすぎない方がいい。
なぜなら、どれだけ綺麗にしても、その服が買えないなら意味が薄い。

あなたのアプリで購買に繋げるなら、必要なのは、
欲しい世界観
↓
買える商品
↓
着た時のイメージ
↓
購入

画像生成の改善を主戦場にしないでください。
今の架空ルック画像は購買判断に弱いです。
STYLE SELFの本命は「実在商品を世界観で選び、その商品を自分または近い体型モデルに着せて、買う判断を助けること」です。
今後は、架空画像生成よりも「商品候補 → 商品画像try-on → 購入導線」を優先してください。
```

## § 2. ★ オーナー最終判断(本セッションでの追加意志)

架空画像生成は「世界観ルック確認」用途として残すと §1 verbatim にあるが、オーナーは本セッション内で「これいらない、ゴミと一緒、無くしてもいい」と ★ さらに強い意志を表明した。

したがって ★ STYLE-SELF の戦略宣言として:

> ★ ★ ★ 架空画像生成は ★ 完全廃止方向 ★ とする。

- 「世界観ルック」UI ボタンも将来的に削除
- 既存の VisualizeButton(C-2g)も廃止対象
- FASHN API の prompt 経路 → 架空生成は ★ 全廃
- プロダクトから完全消去

ただし実装的な削除タイミングは別 Sprint で判断する(★ 戦略 ≠ 即時実装)。本文書は戦略宣言として「廃止」を明示し、実装は順次進める。
