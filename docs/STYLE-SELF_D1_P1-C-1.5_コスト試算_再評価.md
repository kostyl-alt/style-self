## 2. 現実コスト試算(1.5b 完成形・`60c7fa8` 時点)

### 2.1 1 相談あたり実コスト(本体 7.4 + 1.5b 連動)

| 項目 | 内訳(本体 7.4 と同根拠) | 1 往復あたり |
|---|---|---|
| 段階 A | `/api/overlay/intent` Haiku ・入力 ~300 tok + 出力 ~100 tok | ¥0.15 |
| 段階 B | `/api/ai/stylist-chat` Haiku ・入力(system + contextData + history N=3) ~2100 tok + 出力 ~100-200 tok | ¥0.36 |
| **小計(1 往復)** | | **¥0.51** |

### 2.2 複数往復シナリオ(1.5a 以降の現実)

| ケース | 構成 | コスト |
|---|---|---|
| 1 往復で完結 | 段階 A × 1 + 段階 B × 1 | **¥0.51** |
| 平均往復 N=2 | 段階 A × 2 + 段階 B × 2 | **¥1.00**(458c0be 8 章値) |
| 5 往復長会話 | 段階 A × 5 + 段階 B × 5 | **¥2.50** |
| L4-A 切替を 1 回含む(history リセット効果あり)| 切替前後で history=[] なので段階 B 入力 tok が前半往復に等しく戻る → 累積コストは「往復数 × ¥0.51」と同等 | **¥2.50〜¥3.00** |

> 注: L4-A 切替検出は **コスト削減効果あり**(切替時 history=[] でリセットされ、
> 段階 B 入力 tok が膨張しない)。逆に言えば長い会話を切らずに続けると history N=3 の
> slice が効いていても入力 tok は安定 → 1 往復 ¥0.51 が線形に積み上がる。

### 2.3 月想定総コスト(1.5b 完成形時点)

| 想定 | 月相談数 | 平均往復 N | 月額 / ユーザー | 月額 / 1000 アクティブ |
|---|---|---|---|---|
| **本体 7.4 想定**(1 往復モデル)| 150 | 1 | ¥77 | ¥77,000 |
| **1.5a 連動**(diagnose のみ・458c0be 8 章)| 150 | 2 | **¥150** | ¥150,000 |
| **1.5b 累計**(+ closet 100 相談・458c0be 8 章)| 250 | 2 | **¥250** | ¥250,000 |
| **1.5b 長会話混在**(切替含・本 doc 推計)| 250 | 2.5 | **¥300** | ¥300,000 |

→ **¥250-300 / 月 / ユーザー**(本体 7.4 上限 ¥77/月 の **約 3.2-3.9 倍**)

### 2.4 検証期 vs 本格運用期のコスト感

## 4. 将来制限策の選択肢(本体 7.4「将来対応」の具体化)

### 案 P1: 月 N 回上限(ユーザー単位)★ 本命

- **実装**: `app/api/ai/stylist-chat/route.ts` で `users.chat_count_month` + `users.chat_month_key` をチェック・上限超でフォールバック(段階 B スキップ・段階 A の intent-result カード相当を返す)
- **規模**: +20-30 行(route.ts) + DB マイグレーション 1 件(`users.chat_count_month integer`, `chat_month_key text`)
- **UX 影響**: 上限到達時の通知が必要(reply で「今月の上限に達しました。来月から再開できます」)
- **本体 7.4 整合**: ★ 本体 7.1-7.3 の ③ リアル試着 月 N 回制限と **同思想** で実装可能(本体 7.4「将来対応」既述)
- **効果**: コスト天井を確実に管理(1000 人 × N=50 → ¥25,000/月の予算管理可)

### 案 P2: `STYLIST_CHAT_HISTORY_MAX` を N=3 → N=2 に圧縮

- **実装**: `app/(app)/ai/page.tsx:90` の定数を `3` → `2` に変更(1 行)
- **規模**: +0 行(値変更のみ)
- **UX 影響**: 文脈把握が弱まる可能性(リグレッションテスト `3e39f99` ケース c の
  境界値が変わるので連動更新必要)
- **効果**: 段階 B 入力 tok 約 10% 減 → 月額 ~10% 減
- **判定**: 最終手段(品質トレード)

### 案 P3: contextData サマリの要約化

- **実装**: `app/api/ai/stylist-chat/route.ts` の `contextData` 取得側で worldview_profiles 全件 → 上位 3 キーワードのみに圧縮・wardrobe 集計は色系統トップ 3 のみ
- **規模**: +5-10 行(route.ts)
- **UX 影響**: 文脈の細やかさが落ちる(closet 会話の wardrobe 要約の具体性低下)
- **効果**: 段階 B 入力 tok 20-30% 減
- **判定**: 案 P1 と並行投入で更にコスト圧縮可

### 案 P4: クライアント側キャッシュ(同質発話の reply 再利用)

- **実装**: `localStorage` で {発話 hash → reply} mapping(TTL 数時間〜1 日)
- **規模**: +30-50 行(`app/(app)/ai/page.tsx` の handleSubmit 前段に lookup)
- **UX 影響**: 動的会話文脈(sessionIntent / wardrobe 状態変化)で誤キャッシュリスク・
  キーに sessionIntent + messages.length を含める設計が必要
- **効果**: リピート発話の段階 B 呼出ゼロ(ヒット率次第)
- **判定**: ヒット率不明・MVP では効果未知数

### 案 P5: モデル ダウングレード

- **現状**: `claude-sonnet-4-6`(CLAUDE.md ルール)・本体 7.4 試算は Haiku
- **選択肢**: Haiku 系維持・将来の nano 系モデル登場時に再評価
- **効果**: tok 単価変動次第
- **UX 影響**: 会話品質低下リスク(良い例水準超えを `60c7fa8` で達成済のため当面不要)
- **判定**: Anthropic 新モデル登場時の再評価事項(今は触らない)
