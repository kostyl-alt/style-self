## 0. 体験転換の本質

### 現在の方式(navigate 中心・D1-1/D1-2a 実装済)

```
入力(自然言語)
  → /api/overlay/intent で意図判定
  → mode === "navigate" : 既存ページに router.push(画面遷移・離脱)
  → mode === "api"      : 結果を判定結果カードで表示(D1-2b で本実装予定)
  → mode === "hybrid"   : api 結果 + 編集はページ誘導(D1-2c で本実装予定)
  → mode === "none"     : 「準備中」案内

問題:
  - 9 機能(全 18 機能のうち半数)で「結果を見るには画面遷移が必要」
  - 「ボタンが多い・分散している」を解決するはずが、結局画面遷移カタログのまま
  - ChatGPT 型「1 つの対話画面で結果まで完結」とは真逆
```

### 改訂後の方式(対話中心)

```
入力(自然言語)
  → /api/overlay/intent で意図判定(D1-1 のまま温存)
  → 結果を「対話画面の吹き出し / 結果カード」で返す = 基本
  → 専用 UI が本質的に必要な機能だけ「○○しますね →」で滑らかに引き継ぐ

参照: ChatGPT / Claude / Gemini の体験
  - 普通の応答は会話画面内で完結
  - 画像生成・コード編集 等は同画面内のキャンバスに展開(離脱しない)
  - ファイル操作 等の専用 UI が要るものだけ別画面/タブに遷移

= 「離脱を最小化し、対話の中で世界観体験を一貫して提供する」
```

## 1. 18 機能の再分類:対話完結 vs 専用 UI 引き継ぎ

D1-1/D1-2a 調査で得た 18 機能の起動 API/ルート仕様を、**「対話画面で結果まで返せるか」**
の観点で再分類する。

### A. 対話完結群(8 機能)— 結果を対話の吹き出し / 結果カードで返せる

| intent | API | 対話に乗せる形 |
|---|---|---|
| `coordinate` | POST `/api/ai/coordinate` | コーデ JSON を結果カード(画像 + アイテム名 + 理由)で吹き出しに |
| `style-consult` | POST `/api/ai/style-consult` | 相談回答テキストを通常の吹き出しで |
| `inspiration` | POST `/api/ai/abstract-coordinate` | 抽象語→コーデを結果カードで |
| `match-users` | GET `/api/match/users` | 似た人の小さなカード列(display_name + worldview_name + 共通点 N)を吹き出しに |
| `match-posts` | GET `/api/match/posts` | 投稿サムネ列(image + caption + 共通 N)を吹き出しに |
| `culture` | POST `/api/ai/culture-explain` | 音楽/映画/香水の解説テキストを吹き出しに |
| `brand-learn` | POST `/api/brands/recommend` | ブランド推薦カード列を吹き出しに |
| `virtual-coordinate` | POST `/api/ai/virtual-coordinate` | 概念翻訳 + アイテム提案 + 商品の概要を結果カードで(詳細編集だけ navigate) |

**根拠**: いずれも API レスポンスが「ユーザーに見せるべき結果データ」を持っており、
編集や複雑な状態管理が直後に必要ない。Claude Code が画像生成結果を画面内に
出すのと同じ思想で「対話の流れで完結」できる。

### B. 専用 UI 引き継ぎ群(10 機能)— 本質的に専用画面が必要

| intent | 専用 UI 引き継ぎ先 | 理由 |
|---|---|---|
| `diagnose` | `/onboarding` | 16 問の質問フロー + 体型情報入力 + 確認・対話 1 往復で完結不能 |
| `worldview-profile` | `/self?tab=diagnosis` | 自分の世界観の集約画面(複数セクション・編集 UI 含む)|
| `create-post` | `/self/new-post` | 画像アップロード + EXIF/HEIC 処理 + caption・専用フォーム必須 |
| `my-posts` | `/self?tab=posts` | 投稿一覧 + 削除確認モーダル等の管理 UI |
| `closet` | `/outfit?tab=closet` | クローゼットの一覧 + 編集 |
| `saved` | `/saved` | 保存済み一覧(コーデ・商品・投稿の 3 セクション)|
| `history` | `/self?tab=history` | AI 履歴一覧(タイプ別カード描画)|
| `body-edit` | `/self?tab=body` | 体型情報フォーム |
| `preference-edit` | `/self?tab=worldview` | 好み情報フォーム |
| `product-match` | `/outfit?tab=virtual` | 単独叩き不可・virtual-coordinate の延長として専用 UI で扱う |

**対話からの引き継ぎ方**:
```
ユーザー: 「クローゼットを開きたい」
  ↓ 意図判定
アシスタント吹き出し: 「クローゼットを開きますね →」+ [開く] ボタン
  ↓ ボタン押下
専用画面に navigate(モーダル閉じ + router.push)
```

→ 「いきなり画面が変わる」体験を避けて、**対話の中で一拍置いてから引き継ぐ**。
ChatGPT が「ファイルを開きますか?」と確認するのと同じ作法。

### C. 将来機能 2

| intent | 対話完結 / 専用 UI | 備考 |
|---|---|---|
| `moodboard` | **両用**(対話で「ボードを作りますね」→ 作成 UI へ・閲覧は対話に乗せる)| D1-3 で実装 |
| `tryon` | **両用**(対話で「自分の写真を選んでください」→ アップロード UI へ・結果表示は対話)| D1-5 実装(プライバシー専章は不変)|

## 2. 対話 UI の設計方針

### 2.1 OverlayModal を対話画面に拡張

```
現状(D1-2a):
  - モーダル 1 つ
  - 入力欄 1 つ + 1 回送信 → 結果 1 つ表示
  - 履歴なし

改訂後:
  - モーダル(or 全画面)を対話画面に拡張
  - 上部: メッセージ履歴(吹き出し列・ユーザー / アシスタント交互)
  - 下部: 入力欄(常時表示・連続発話可能)
  - 結果は吹き出し内に結果カード(コーデ / 似た人 / 投稿 / カルチャー 等)
```

### 2.2 全画面化の判断(★ オーナー判断必要)

| 案 | 評価 |
|---|---|
| **(A) 全画面オーバーレイ** | ChatGPT / Claude Code 型・対話に集中できる・BottomNav 非表示 |
| **(B) 現状モーダル拡張** | BottomNav 共存可・モーダル内が狭い・履歴表示量に制約 |

→ **オーナー判断材料**: ChatGPT 型を目指すなら (A)・「既存 BottomNav も自然に共存」したいなら (B)。
本改訂案では **(A) 全画面オーバーレイ** を推奨(対話中心の本質に最も近い)。
BottomNav は対話を閉じれば常時アクセス可能・離脱は対話の終了で表現される。

### 2.3 結果カードの設計(対話完結 8 機能)

各 intent ごとに「結果カード型」を定義:

| intent | カード型 | 主要表示要素 |
|---|---|---|
| `coordinate` / `inspiration` | コーデカード | 各アイテム名 + 色 + 素材 + 理由 + (将来) シルエット図 |
| `match-users` | 人マッチ列 | display_name + worldview_name + 共通点 N + `/u/[id]` リンク |
| `match-posts` | 投稿サムネ列 | 画像 + caption(短) + 共通 N + `/p/[id]` リンク |
| `culture` | カルチャーカード | music / films / fragrance を 3 セクション |
| `brand-learn` | ブランドカード列 | brand_name + 哲学 + worldview_tags(日本語表現で) |
| `style-consult` | テキスト吹き出し | 通常会話 |
| `virtual-coordinate` | コンセプト + コーデ + 商品概要 | 概念翻訳 + items + 商品 TOP3(★worldview_tags 露出ゼロ・M4 同型点検)|

**プライバシー要件**(本体設計書 4.4 同等):
- worldview_tags 英語スラッグは結果カードに **絶対露出させない**
- 「共通点 N 個」等の抽象表現で抽象化(M4 / M5-3 確立)
- View Source 漏洩点検を改訂版実装の各段階で実施

### 2.4 専用 UI 引き継ぎ群の対話表現(10 機能)

```
ユーザー: 「投稿作りたい」
  ↓ /api/overlay/intent → intent=create-post / mode=navigate
アシスタント吹き出し:
  「新しい投稿を作りますね。画像と一言を準備してください」
  [投稿作成画面を開く →]   ← ボタン押下で router.push("/self/new-post")
```

→ navigate-map(D1-2a 資産)は **完全に転用**。捨てない。
→ ?tab= バグ修正(A の最小修正)はこの引き継ぎを前提に **必須**。

### 2.5 会話文脈(state 管理)の方針

**MVP 範囲(★ オーナー判断必要)**:

| 案 | 内容 |
|---|---|
| (i) 単一セッション内 React state 履歴のみ(永続化なし) | モーダル/全画面オーバーレイを閉じると履歴消滅。「さっきの〜」はオーバーレイを開き続けている間だけ動く |
| (ii) localStorage で軽量永続化 | refresh しても短期間履歴が残る・privacy 観点で容量制限要 |
| (iii) DB に保存(ai_history 等を活用)| 既存履歴機能と統合できる・実装重い |

→ **本改訂案推奨**: **(i) 単一セッション内 state**(MVP)。
- 対話中心の本質的 UX を最低限満たす
- 「さっきの理想コーデの商品見せて」は同セッション内なら state に items が残っている前提で動く
- 永続化は D-1 後半 or 将来

### 2.6 「さっきの〜」を活かす連鎖の例

```
ユーザー: 「Yohji の世界観でコーデ作って」
  ↓ intent=virtual-coordinate
アシスタント: [コンセプト翻訳 + items + 商品概要カード]

ユーザー: 「これに合う商品もっと見せて」
  ↓ intent=product-match(★ items を直前の virtual-coordinate 結果から引継)
アシスタント: [商品 TOP3 のさらに詳細]

ユーザー: 「似た世界観の人は?」
  ↓ intent=match-users
アシスタント: [人マッチ列]
```

→ MVP は **virtual-coordinate → product-match 連鎖** だけ実装(D1-2a 調査で
「product-match は単独叩き不可」と確定済のため・連鎖が本質的に必要)。
他の「さっきの〜」は将来。

## 5. 地雷リスト(対話中心化で新規 + 既存の更新)

| # | 地雷 | 対策 |
|---|---|---|
| 1 | 履歴 state の肥大化(長い対話で結果カード多数)| 履歴は単一セッションのみ(永続化なし)・ resultCard 数を最新 N 件で表示制限 |

| 10 | 一度に作り変える危険 | **段階分割厳守**(D1-2x → b' → c' → d' → e')・既存 D1-1/2a の OverlayModal を段階的に拡張・一気に書き換えない |
