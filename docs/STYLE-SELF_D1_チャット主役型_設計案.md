作成日: 2026-05-20
位置づけ: 本ドキュメントは **再設計"案"**。本体設計書
          `docs/STYLE-SELF_D1_実装設計.md`(現 HEAD 8b8bc2a・対話中心改訂版)は
          **まだ書き換えない**。オーナー合意後に本体へ再改訂する前提の議論材料。
背景: D1-2b' 完了(`483cef4`)後、オーナーから根本方針転換と 2 ドキュメント提示。
      「アプリを開いたら最初に出るのはチャット画面・FAB 廃止・既存タブを

## 0. 転換の本質(オーバーレイ型 → チャット主役型)

### 旧:オーバーレイ型(現 本体設計書 8b8bc2a・実装は 483cef4 まで)

```
アプリ起動 → /home(ホーム画面)
BottomNav 5 タブ: home / discover / saved / outfit / self
右下 FAB(黒い吹き出しボタン)→ タップで OverlayModal 展開 → 対話画面
チャットはあくまで「追加機能」・既存 18 機能のカタログは残ったまま
```

### 新:チャット主役型(本案・MVP の最終形)

```
アプリ起動 → チャット画面(/ または /ai)
BottomNav は 3 タブに集約: AI(チャット)/ 保存 / 自分
FAB は廃止(チャットそのものがメイン)
既存 18 機能はチャットから "裏で" 呼ぶ(API は完全温存・導線のみ変更)
ユーザーは機能名を選ばず自然文で相談
  「この服に何合わせれば」「自分に着せて」「ムードボードっぽくして」
AI 返答は文章 + カード形式(結論 / 合う理由 / 合わない可能性 / 代替案 / コーデ案 / 次アクション)
```

### なぜ再転換か(オーナー実機 FB の記録)

```
対話中心改訂(D1-1/2a/2b')は「対話モーダルが既存タブの上に被さる」形だった。
オーナー実機で:
  ・FAB + 5 タブ並列は「ChatGPT を起動するボタンを置いた普通アプリ」
  ・タブで機能を探す導線が残るため、「自然文で全部できる」体験になっていない
  ・ChatGPT の服版を目指すなら "チャットがメイン画面そのもの" でなければならない

→ オーバーレイ型は機能としては動くが、目指す体験ではなかった。
   対話中心改訂はあくまで対話 UI の土台確立工程として価値があり、
   その資産(intent ルーター / navigate-map / 対話ロジック)は
   チャット主役型でも完全に転用できる(後述セクション 8)。
```

### 理想(ChatGPT の服版)

```

入力 → 自然文 → 意図判定 → 既存 API を裏で呼ぶ → カード形式で回答

自分タブ → プロフ管理のみ(世界観 / 身体 / 好み / 避けたい印象 / 履歴 / 設定)
```

## 4. AI 返答のカード形式設計

### 4.1 標準カード型(全機能で共通の骨格)

```
┌────────────────────────────────────────┐
│ [見出し]   例:「黒コートのコーデ提案」 │
├────────────────────────────────────────┤
│ [結論]     例:「上はチャコールニットが合います」│
│ [合う理由] 例:「ウールの落ち感同士で素材調和」│
│ [合わない可能性] 例:「光沢が強いと世界観外」│
│ [代替案]   例:「ベージュチェスターも可」    │
│ [コーデ案] [アイテム ×N(画像 + 名前)]    │
├────────────────────────────────────────┤
│ [次アクション]                          │
│ [💾 保存] [🛍 商品を探す]              │
│ [👗 着せてみる] [🔄 別案]              │
└────────────────────────────────────────┘
```

### 4.2 機能別カードのバリエーション(対話中心改訂セクション 4.2 と整合)

| intent | カード型 | 主要要素 |
|---|---|---|
| `coordinate` / `inspiration` | コーデカード | 各アイテム名 + 色 + 素材 + 結論 + 理由 + コーデ画像(将来)|
| `virtual-coordinate` | コンセプト + コーデ + 商品 TOP3 | 概念翻訳 + items + 商品(連鎖 product-match)|
| `match-users` | 人マッチ列 | display_name + worldview_name + 共通点 N + `/u/[id]` リンク |
| `match-posts` | 投稿サムネ列 | 画像 + caption(短)+ 共通 N + `/p/[id]` リンク |
| `culture` | カルチャーカード | music / films / fragrance の 3 セクション |
| `brand-learn` | ブランドカード列 | brand_name + 哲学 + worldview_tags(日本語表現で)|
| `style-consult` | テキスト吹き出し | 相談回答 |
| `product-match` | 商品カード列 | 商品名 + ブランド + 価格 + 「世界観 +N」バッジ |

### 4.3 D1-2a 既存 5 サブ(ResultView / NavigateConfirm / NoneNotice / SuggestionList / ApiHybridPlaceholder)の発展

```
旧: モーダル内の判定結果分岐(D1-2a / D1-2b' で吹き出し化済)
新: ChatPage 内の「アシスタント吹き出し」の中で同じ役割を担う
    + カードを D1-2c' で各 intent ごとに専用化(セクション 4.2)

破棄せず発展:
  - ResultView: チャット主役でも判定結果の親分岐として継続
  - NavigateConfirm: 「○○しますね →」+ navigate-map の URL 表示で継続
  - NoneNotice: moodboard / tryon / unknown 案内で継続
  - SuggestionList: confidence 低時の候補提示で継続
  - ApiHybridPlaceholder: D1-2c' で各 intent の専用カードに置換していく
```

### 4.4 「次アクション」ボタン群(チャット主役型の核)

```
ユーザーが機能名を選ばず、AI 返答カードから「次に何をするか」を選べる:
  [💾 保存]         → 既存 /saved に追加(M3 既存 API 経由)
  [🛍 商品を探す]    → product-match 連鎖(virtual-coordinate 結果から)
  [👗 着せてみる]    → tryon(Phase 3-4・プライバシー設計後)
  [🔄 別案]          → 同 intent で再生成
  [📌 ムードボードへ]→ moodboard(Phase 2)
  [👥 似た人を見る]  → match-users
```

これが「対話完結 + 連鎖」の本質。Phase 1 では [保存] [商品を探す] [別案] の 3 つを実装。
[着せてみる] [ムードボードへ] は Phase 2 / 3 で追加。

## 5. クレジット消費を抑える設計(M5-4a 試算作法を踏襲)

### 5.1 モデル選定(階層化)

| AI 呼出 | モデル | 1 回コスト | 用途 |
|---|---|---|---|
| 意図判定(/api/overlay/intent)| Haiku 4.5 | **¥0.001** | 全入力に対し必須・最軽量 |
| 自然文短返答(吹き出し内テキスト)| Haiku 4.5 | ¥0.001-0.003 | 対話文・「○○しますね」程度 |
| コーデ提案(/api/ai/coordinate)| Sonnet 4.6(既存)| ¥0.01-0.03 | 構造化 JSON + 説明 |
| 着こなし相談(/api/ai/style-consult)| Sonnet 4.6 | ¥0.01-0.03 | 対話的回答 |
| 理想コーデ(/api/ai/virtual-coordinate)| Sonnet 4.6 + Knowledge OS | ¥0.03-0.08 | 多段(Stage 1-3 + 知識 lookup)|
| カルチャー解説(culture-explain)| Sonnet 4.6 + キャッシュ 30 日 | ¥0.02 / 初回のみ | 既存 30 日 localStorage キャッシュ活用 |
| **③ リアル試着(virtual try-on)** | 外部 API | **¥10-30** | Phase 3-4・コスト管理章不変 |

### 5.2 呼び出し最小化策

```
1. 提案チップ選択時は意図判定スキップ
   → 6 種チップは intent 直接付与で /api/overlay/intent を呼ばない(Haiku 1 回節約)

2. 既存キャッシュの活用
   → culture-explain は localStorage 30 日キャッシュあり(Sprint 48 既存)
   → 同コンセプトでの再相談は1日内ならクライアント側キャッシュ検討(将来)

3. 直近の virtual-coordinate 結果を state 保持(対話中心改訂 判断5-③ MVP 含む)
   → product-match を「さっきのコーデで商品探して」で再呼出時、items を state から
     引き継いで /api/ai/virtual-coordinate を再実行しない(Sonnet 1 回節約)

4. 連投レート制限(クライアント側)
   → 送信ボタン disabled 中は次送信不可(D1-2b' 既存)
   → さらに 2-3 秒の最小間隔を入れる(将来検討)

5. 「考えています…」中の入力は無効化
   → loading 中の連投を完全に防ぐ(D1-2b' 既存・継続)
```

### 5.3 コスト概算(1 ユーザー / 月)

```
想定: アクティブユーザー が 1 日 5 相談 × 月 30 日 = 月 150 相談

内訳(チップ選択 半分・自然文 半分 と仮定):
  ・意図判定 Haiku   : 75 回 × ¥0.001 = ¥0.075
  ・自然文短返答     :150 回 × ¥0.002 = ¥0.30
  ・コーデ提案 Sonnet: 50 回 × ¥0.02  = ¥1.00
  ・相談・解説       : 50 回 × ¥0.02  = ¥1.00
  ・virtual+product  : 30 回 × ¥0.04  = ¥1.20
  ・カルチャー(キャッシュ後): ¥0.10

→ 1 ユーザー月 約 ¥3.7(★ ③ リアル試着除く・Phase 1 範囲)
  100 アクティブで月 ¥370 / 1000 アクティブで月 ¥3,700

③ リアル試着(Phase 3-4)の見積もりは本体設計書セクション 7 のまま不変:
  - 月 10 試着 × ¥10-30 = ¥100-300 / ユーザー
  - 100 アクティブ × 月 10 試着 = ¥10,000-30,000 / 月
  → ★利用回数制限(月 N 回・本体設計書 7.1 維持)で抑制
```
