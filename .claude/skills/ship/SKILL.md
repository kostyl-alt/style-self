# /ship — style-self 実装フロー

⚠️ このスキルは **style-self/ を cwd にして起動している前提**。違うなら冒頭で報告して止まる。

## ステップ0（最優先）: コンテキスト実ロード確認

実装に入る前に、CLAUDE.md と該当 rules が**実際に読み込まれているか**を確認する。
- CLAUDE.md が読まれているか（プロジェクト憲法・モデル使い分け・「構造で強制」思想を把握しているか）。
- これから触る領域の path-scoped rules が注入される設計か。例: `lib/prompts/**` や `app/api/ai/**`、`lib/utils/vision-analyzer.ts` を触るなら `styling-quality.md` が注入される想定。
- 確認方法: 対象ファイルを Read した直後に rule body が system-reminder で注入されるか。不安なら `/context`・`/memory` でユーザーに確認を促す。
- **読まれていない/注入されないなら、実装に進まずその事実を報告して止まる**（推測で進めない）。

## ステップ1: 要件再定義
ユーザーの依頼を1〜2文で言い換え、「何を・なぜ・完了条件」を確定。曖昧なら必要時のみ確認（必要時以外は止まらない）。

## ステップ2: 調査
関連ファイル・既存パターン・横断フロー（チャット2段パイプライン / ブランドマッチ / MB分析3層 等）への合流点を把握。

## ステップ3: 影響範囲
変更が波及する消費者（読み手）を特定。「additive で消費者ゼロか / 既存挙動が変わるか」を明示。

## ステップ4: 最小設計
最小差分の設計を1案。フラグは default OFF・OFF で現状維持＝回帰ゼロを原則。

## ステップ5: 必要時のみ確認
設計が複数解で本人判断が要るときだけ確認。それ以外は進む。

## ステップ6: 実装
Style-Self 鉄則（下記）に沿って実装。`as never` キャスト・`"use client"` 最小・Claude 呼び出しは `app/api/` から等の既存パターン踏襲。

## ステップ7: 検証（基本）
package.json の scripts を確認し、基本検証は `npx tsc --noEmit` → `npm run lint` → `npm run build`。
lint script が無い/壊れている場合は事実として報告し、勝手に新設しない。
専用テストランナーは無く、test 整備は将来の別タスク。

## ステップ8: 動作確認観点
認証が要るチャット/MB系はコードだけでは検証不能。「何を実機で見れば確認できるか」を観点として明示。

## ステップ9: 差分確認
`git diff` で意図しない変更が無いか確認。**勝手に commit しない**。

## ステップ10: 初心者向け報告（フォーマット固定）
```
## 変更したこと
（何を・なぜ。専門用語は噛み砕く）
## 触ったファイル
（パス + 一言）
## 確認したこと
（tsc / lint / build の結果を実際の出力で）
## 残っているリスク
（未検証点・前提）
## 私が実機で確認すること
（オーナーが見るべき画面と観点）
## 次にやるなら何か
（任意の次手）
```

## ステップ11: 実機チェックリスト
コミット前に必要な実機確認項目を箇条書きで提示。**オーナーの実機OKを待つ**。

## Style-Self 鉄則（見出しのみ・詳細は `styling-quality.md`）
- 構造で強制（祈らない）／LLM が盛る余地はコードで消す
- 事実ベース・写っていないことは断定しない（`basis: observed|inferred`）
- 固有名・国籍は捏造しない（ブランドカードは `brand-match.ts` が組む）
- モデルは原則 Sonnet・軽量のみ Haiku（claude.ts の例外）
- `any` 禁止 / `console.log` 残置禁止 / Supabase 操作は `lib/` 経由

## コミット/プッシュ規律
- 勝手に commit しない・commit 前にユーザー確認を挟む（`co-authored-by` 不要）
- **push 禁止**（明示依頼があるまで）

## 進め方
結論は短く。止まらず進む。確認は必要時のみ。各ステップの結論は短く記録し、ユーザー確認が必要な場合だけ止まる。
